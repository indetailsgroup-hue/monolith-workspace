# Implementation Plan: Capture Spine (Monolith, Phase 2)

## Overview

แผนการสร้าง Capture Spine — pipeline วงปิด ingest → OCR → extract → verify → commit + audit โดย **ลอก lifecycle TCCK `agent_artifact`** และ **reuse-not-fork** C12 / workflow-copilot (human verify/approval) / MCP governance (R15-19) / line-oa / Typhoon OCR self-host

แนวทาง: **DB-first → SECURITY DEFINER RPC → RLS via C12 → append-only audit → property-based test → Edge Function (on-prem)** บน Supabase/PostgreSQL + Deno Edge Functions; pure logic ใน `src/capture/` เพื่อ PBT แบบ deterministic

> **Discipline:** human verify invariant (ไม่ commit จนกว่ามนุษย์ verify); fail-safe ไม่เดาค่า (no placeholder); on-prem boundary (no outbound นอก org); reuse-not-fork; config-driven per-department
>
> **OPEN GATE-1:** migration ใหม่ของ capture-spine ต้องรัน `supabase db reset` เขียวก่อนถือ task done — **supabase CLI 2.108.0 + Docker พร้อมในเครื่องนี้** (C12+line_oa substrate เขียวแล้ว verified)
>
> **DEP:** Phase 2 — พึ่ง `monolith-workflow-copilot` (Phase 1: human verify/approval/SLA) + `monolith-mcp-layer` (D2/audit/rate-limit/idempotency) + C12/`line_oa_customer_identity` (deploy-verified) + verify_rules seed จาก `_knowledge-export.json`

การทดสอบ: **Vitest + fast-check** (Property 1–11 → 1 property test/ข้อ, ≥ 100 iteration, tag `// Feature: capture-spine, Property N: ...`), **pgTAP/DB-harness** สำหรับ RLS/immutability/idempotency, **smoke** สำหรับ on-prem boundary; งานทดสอบเป็น sub-task ทางเลือก (`*`)
