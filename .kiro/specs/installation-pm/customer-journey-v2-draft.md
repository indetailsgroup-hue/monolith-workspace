## เฟส 1 · Add OA → Qualify → เปิดบ้านในระบบ

| ก้าว | ตำแหน่งที่เกี่ยว (จาก JD จริง) | ระบบทำ |
|------|-------------------------------|--------|
| ลูกค้า add OA → คุย | **H2 เจ้าหน้าที่ฝ่ายขาย (Sales Staff)** — พบลูกค้า/เก็บความต้องการ; **H1 ผจก.ขาย** คุมทีม+เป้า | conversation + ตัวตนลูกค้าเกิดอัตโนมัติ (มีแล้ว) |

## เฟส 3 · ออกแบบ — จับคู่ designer กับความชอบลูกค้า

| ก้าว | ตำแหน่ง | ระบบทำ |
|------|---------|--------|
| **จับคู่ designer ตามสไตล์ลูกค้า** | **B1 ผจก.ฝ่ายออกแบบ** = ผู้ approve การจับคู่ (JD: คุมทีมออกแบบ + = Gate keeper ตาม ADR-031); ระบบเสนอ **B2 Designer** จาก Mood&Tone ในใบ requirement ↔ ความถนัด/portfolio ของ designer | ระบบเสนอ → B1 approve → B2 เข้ากลุ่มที่ 1 → แนะนำตัวกับลูกค้า |
| ออกแบบ + นำเสนอ | B2 (JD: พบลูกค้า กทม./ตจว., ศึกษาโครงสร้าง+งบ+วัสดุ, นำเสนอแบบ+ปรับแก้) + **B3 เจ้าหน้าที่ 3D Rendering** (3D model + rendering + material setting) | อนุมัติแบบ G1/G2/G3 ผ่านการ์ดในกลุ่มที่ 1 (approval loop เดิม — ลูกค้าอยู่ใน quorum) |
| **ตรวจหน้างานร่วมก่อนผลิต** (ขั้นใหม่จาก owner) | **C2 ทีมวัด + B2 Designer ไปบ้านลูกค้าด้วยกัน** — ตรวจทุกรายละเอียดของแบบกับหน้างานจริง → **ลูกค้า approve** | บันทึกผลตรวจ (capture) + การ์ดอนุมัติของลูกค้า = ประตูสู่การผลิต |

## เฟส 4 · เข้าผลิต — เปิด "กลุ่มที่ 2 = กลุ่มโรงงานของบ้านนี้"

| ก้าว | ตำแหน่ง | ระบบทำ |
|------|---------|--------|
| รับงานเข้าแผนผลิต | **B4 เจ้าหน้าที่วางแผนการผลิต (= "factory planner")** — JD: แปลงแบบเป็นแบบผลิต (Pytha→MaxCut), **คำนวณราคา**, ทำรายการวัสดุ→คลัง/จัดซื้อ, ไฟล์เจาะ+checklist; ร่วม **B5 Draftsman** (ไฟล์ตัด/เจาะ) | ระบบ add B4 เข้า workflow (ขั้น Production Planning เดิม — G4 internal gate) |
| **สร้างกลุ่มที่ 2** | สมาชิก: **B2 Designer · B4 Planner · ผู้ตรวจ BOM ฝั่งโรงงาน (E2 หัวหน้าฝ่ายผลิต — วางแผนวัตถุดิบ/ตารางผลิต + E6 คลัง — เบิก-จ่ายวัสดุ) · ทีมโรงงาน** — **ไม่มีลูกค้า ไม่มี sale ไม่มีทีมวัด** | group_type ใหม่ `factory` (ปัจจุบันมีแค่ internal/customer — ต้องขยาย 0095) + guardrail เดิม: ข้อมูลกลุ่มนี้ไม่มีทางไหลเข้ากลุ่มลูกค้าอัตโนมัติ |
| ผลิตรายสถานี + รายงานเข้ากลุ่ม 2 | **E3 Production Operator 6 สถานี** (Laminate HPL → Cutting → Edging → CNC → Assembly → Packing) ส่งรูป/รายงานจบสถานีเข้ากลุ่ม; **E2** คุมตาราง; **E5 QA** ตรวจระหว่างผลิต | bot เก็บรูปต่อสถานีเป็นหลักฐาน (pattern เดียวกับรูปจบเลนติดตั้ง) |
| **Designer approve ต่อสถานี → ลูกค้าเห็นเป็นระยะ** | **B2/B1** approve รูปงานที่จบขั้น (JD B1: "ตรวจชิ้นส่วนหลังผลิตตรง spec") | รูปที่ approve แล้ว → ระบบส่งเข้า**กลุ่มที่ 1** เป็น curated update อัตโนมัติ — **นี่คือคำตอบของ "ช่วงเงียบ"**: ลูกค้าเห็นตู้ตัวเองค่อย ๆ เกิดขึ้นทีละสถานี |
| ผลิตเสร็จทั้งหมด | B2 approve รวมทั้งหมด + **E5 QA ด่านสุดท้ายก่อนออกจากโรงงาน** (JD: อำนาจอนุมัติของดี/ของเสีย) | สถานะ → พร้อมจัดส่ง |

## เฟส 5 · จัดส่ง → ติดตั้ง → QC → ส่งมอบ

| ก้าว | ตำแหน่ง | ระบบทำ |
|------|---------|--------|
| แจ้งจัดส่ง | **E7 หัวหน้าฝ่ายโลจิสติกส์** — วางคิว (packet + ทีม + รถ + เวลานัด), คุมโหลด/ตรวจนับก่อนออก (JD = จุดเชื่อม Factory→Installation) | ระบบส่ง**แผนติดตั้งฉบับลูกค้า**เข้ากลุ่มที่ 1: วันเริ่ม–วันเสร็จ–วันส่งมอบ + เอกสารการทำงาน |
| ทีมติดตั้ง+QC เข้ากลุ่มที่ 1 | **D4 หัวหน้าทีมติดตั้ง** (1 คน/บ้าน — approver เริ่ม/จบ) + ทีม (D5/D6/D7 ตามโครง 3 ช่าง/ห้อง) — อนุมัติการจัดทีมโดย **D3 หัวหน้าฝ่ายติดตั้ง** (JD: คุมทุกทีม) ภายใต้ **D1 PM** (จัดสรรทรัพยากรต่อโครงการ + รับ escalation) · **E5 QA** เข้าตรวจที่บ้าน | roster เฟสติดตั้ง → D4 เข้ากลุ่มลูกค้า (ช่างอยู่เฉพาะกลุ่มงานภายใน/ไม่เข้ากลุ่มลูกค้า — ตามมติเดิม); T0 → ติดตั้งรายเลน → ใบปิดบ้าน (มีแล้วทั้งเส้น 0090–0097) |
| ตรวจรับ | ลูกค้ากดการ์ด Flex (มีแล้ว 0098) — **เพิ่ม: QC (E5) ตรวจก่อนส่งการ์ด** = สองชั้น: QC ภายในผ่านก่อน แล้วค่อยเชิญลูกค้าตรวจรับ | completed → after-sales (เฟสถัดไป — ยังไม่ออกแบบ) |

## สิ่งที่ v2 เพิ่มจากระบบที่มี (delta ที่ต้องสร้าง)

1. **Phase Roster + Assignment Approval** — ตาราง "ใครควรอยู่กลุ่มไหนช่วงไหน" ต่อบ้าน + คำขอ assign ที่หัวหน้าฝ่ายนั้น approve (C1 วัด, B1 ออกแบบ, D3 ติดตั้ง, E2 โรงงาน) → ระบบแจ้งคน + ตามคนเข้ากลุ่ม + เตือนคนที่ควรออกเมื่อจบเฟส
2. **group_type `factory`** (กลุ่มที่ 2) + กติกาสมาชิก/guardrail ของมัน
3. **Designer matching** — เสนอ B2 จาก Mood&Tone ↔ ความถนัด (เริ่มจาก manual: B1 เลือกจาก list ที่ระบบจัดเรียงให้)
4. **Joint site verification** ก่อนผลิต (C2+B2 ที่บ้าน + ลูกค้า approve) — เป็น capture type ใหม่หรือ gate ใหม่?
5. **Production milestones รายสถานี** + Designer approve + **auto-curated ไปกลุ่มลูกค้า** (โครงเดิมรองรับ: capture + template audience=customer + guardrail — ขาด milestone model)
6. **QC (E5) gate ก่อนเชิญลูกค้าตรวจรับ** — สองชั้น: ภายในผ่าน → ลูกค้าค่อยตรวจ
7. แผนติดตั้งฉบับลูกค้า (วันเริ่ม/เสร็จ/ส่งมอบ + เอกสาร) ส่งเข้ากลุ่มอัตโนมัติ — จาก E7 คิวส่ง + D1/D2 แผนโครงการ

## ⚠️ ข้อจำกัดแพลตฟอร์มที่ต้องออกแบบรอบ (ความจริงของ LINE)

**LINE ไม่อนุญาตให้ bot สร้างกลุ่มหรือดึงคนเข้ากลุ่มเอง** (Messaging API ไม่มี invite) — "ระบบ add อัตโนมัติ" จึงต้องแปลเป็น **"ระบบสั่ง-เช็ค-ตาม"**:
ระบบรู้ roster ต่อเฟส → push แจ้งคนที่ถูก assign ("คุณถูกมอบหมายเข้าบ้านคุณสมชาย — ให้หัวหน้า/Sale เชิญเข้ากลุ่ม") → คนในกลุ่มกดเชิญ (30 วินาที) → bot เห็น join → match roster → ติ๊กครบ/ตามคนขาด/แจ้งคนเกิน — ผลลัพธ์ต่อผู้ใช้เหมือน "อัตโนมัติ" (ไม่ต้องจำว่าใครต้องอยู่ ระบบจำแทน) โดยไม่ฝืนแพลตฟอร์ม

## มติ grill journey-v2 (owner, 7 ก.ค. 2026)

> **มติ 1 — กลุ่มโรงงาน = กลุ่มถาวรกลุ่มเดียว** (ไม่เปิดต่อบ้าน): สมาชิกประจำ E2 + ตัวแทน 6 สถานี + E5 QA + E6 คลัง (+ B2/B4 ของบ้านที่กำลังผลิต); bot แยกงานต่อบ้านด้วยชื่อบ้านนำหน้าทุกโพสต์ + รูปผูกบ้านด้วยรหัส; เหตุผลชี้ขาด: **กลุ่มตามธรรมชาติของทีม** — ทีมหมุนตามบ้าน (วัด/ติดตั้ง) ใช้กลุ่มต่อบ้าน, ทีมอยู่กับที่ (โรงงาน) ใช้กลุ่มประจำ; ประวัติต่อบ้านไม่เสียเพราะหลักฐานจริงถูก capture แยกบ้านในระบบ (กลุ่ม = ช่องทางคุย ไม่ใช่ที่เก็บความจริง); ผลต่อ schema: group_type เพิ่ม 'factory' แบบ**ไม่ผูก project** (กลุ่มระดับบริษัท) — ต่างจาก internal/customer ที่ผูกบ้าน

> **มติ 2 — Designer gate 2 จุด, รายงาน 6 จุด**: ทุกสถานีส่งรูปเข้ากลุ่มโรงงาน (FYI — หลักฐานครบ + designer ทักได้ทันทีแบบ andon ไม่บังคับรอ; E5 QA คุมชั้นในตาม JD); **Gate 1 = หลัง Assembly** (designer approve → รูปตู้จริง curated เข้ากลุ่มลูกค้า) · **Gate 2 = หลัง Packing** (designer approve รวม + E5 ด่านสุดท้าย → แจ้งจัดส่ง); ลูกค้าเห็น 3 จังหวะ: "เริ่มผลิตแล้ว" (อัตโนมัติ) → "ตู้เสร็จแล้ว 📷" → "ขึ้นรถแล้ว 🚚"; หลัก: **หลักฐานเก็บถี่ gate ตั้งเฉพาะจุดที่การหยุดรอคุ้ม** — Assembly คือจุดแก้ถูกครั้งสุดท้าย

> **มติ 3 — ตรวจหน้างานร่วมก่อนผลิต = เงื่อนไขก่อนส่งการ์ด G3 (ไม่ใช่ gate/ขั้นใหม่)**: capture ใหม่ `site_design_verification` — C2 ทีมวัด + B2 designer ตรวจแบบเทียบบ้านจริงร่วมกัน (รูป+ผลตรวจ) → มี capture verified แล้ว การ์ดเซ็นแบบ final (G3) ค่อยถูกส่ง → **ลายเซ็น G3 = อนุมัติหลังเห็นของจริง** (เถียงไม่ได้ตอน dispute); ไม่แตะ canonical FSM/RACI; enforcement เริ่ม soft+audit (pattern T0 — ADR-039 มติ 4) ยกเป็น hard ที่จุด handoff ได้ภายหลัง; ปัดตก: ขั้นที่ 9 ใน canonical (ผ่าตัดใหญ่เกินประโยชน์) และ subject ใหม่ใน installation_approvals (reopen ADR-039 มติ 1)

> **มติ 4 — แผนติดตั้งถึงลูกค้า: E7 จัดคิว → ระบบร่าง → D1 PM กดยืนยันส่ง (คลิกเดียว)**: ข้อมูลจากคนรู้จริง คำสัญญาออกจากเจ้าของโครงการ; snapshot ทุกฉบับลง audit + เก็บ version (เลื่อนกี่ครั้งวัดได้); หลัก: การกระทำที่ผูกมัดภายนอกต้องมีเจ้าของมนุษย์เสมอ — pattern เดียวกับใบปิดบ้าน/ส่งตรวจรับ

> **มติ 5 — การเงินในระบบเฟสแรก = แจ้ง + บันทึก + soft gate**: ระบบเก็บแผนงวดต่อบ้าน (งวด/%/ผูกเหตุการณ์: เซ็นสัญญา · เซ็นแบบ final G3 · ก่อนส่งมอบ-ตรวจรับ) → ถึงจังหวะส่งการ์ดแจ้งงวดเข้ากลุ่มลูกค้าอัตโนมัติ (ยอด+บัญชีโอน) → F3 เจ้าหน้าที่การเงินเช็คยอดเข้าแล้วกดบันทึกรับ → gate แบบเตือน: ปล่อยผลิตทั้งที่งวดก่อนผลิตไม่เข้า = ระบบเตือน + ต้องมี D1 PM/GM override พร้อมเหตุผลลง audit (ไม่ hard block — exception มีจริงเสมอ, hard gate = ไล่คนออกนอกระบบ บทเรียน T0); ไม่ต่อ payment gateway (รับโอนแบงก์ตามจริง); เป้าหมายจริง: ไม่มีงวดถูกลืม + ไม่มีการปล่อยผลิตโดยไม่รู้ว่าเงินยังไม่เข้า; **รอ seed: โครงงวดมาตรฐาน DAPH (กี่งวด/กี่%/ผูกจุดไหน)**

> **มติ 5.1 — โครงงวดมาตรฐาน DAPH: 4 งวดหน้าหนัก งวดส่งมอบ 5%** — จุดผูก 4 จุดตาม journey: (1) เซ็นสัญญา/มัดจำ (2) เซ็นแบบ final G3 → ก่อนผลิต (3) ก่อนจัดส่ง (Gate 2 Packing ผ่าน/แจ้งจัดส่ง) (4) ตรวจรับ = **5%**; สัดส่วนสามงวดแรก = ค่า default ระดับระบบที่ปรับได้ต่อสัญญา — **default ยืนยันแล้ว (owner, 7 ก.ค. 2026): 50/30/15/5** — 50% เซ็นสัญญา/มัดจำ · 30% เซ็นแบบ final G3 · 15% ก่อนจัดส่ง · 5% ตรวจรับ; งานโครงการ/corporate ใช้โครงต่างได้ (payment plan เป็นของต่อบ้าน ไม่ hardcode)
