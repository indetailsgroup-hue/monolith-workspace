03 Gating mechanics gate มี 3 แบบเท่านั้น — เก็บเป็น gate_kind enum ใน DB
boolean
เปิด/ปิดฟีเจอร์ตรง ๆ เช่น export เครื่องยุโรป (P2P native) ตรวจก่อนเปิดเมนูหรือก่อน action
เก็บ: bool_value  ·  ตรวจ: has_feature()
stock_quota
นับ "ของที่มีอยู่ ณ ปัจจุบัน" เทียบ limit เช่น projects, seats, cloud storage — ลดลงเมื่อลบ ไม่ reset ตามรอบ
เก็บ: limit_value + นับ row จริง  ·  ตรวจ: trigger ที่ DB
metered_quota
นับ "การใช้ต่อรอบบิล" เช่น AI design assist ต่อเดือน — reset ต้นรอบใหม่
เก็บ: usage_counters ต่อ period  ·  ตรวจ: can_consume()
บังคับ 3 ชั้น: UI — ซ่อน/disable→ API / Edge Function — กันก่อนทำ→ DB — RLS + trigger (ด่านสุดท้าย)
04 Supabase schema แยก plan ออกจาก entitlement → ทำ grandfathering ผ่าน override ได้โดยไม่สร้าง plan ใหม่
auth.users ──1:1── profiles │ organizations ──1:*── memberships (user ↔ org, role) │ ├──1:1── subscriptions ──*:1── plans ──1:*── plan_entitlements ──*:1── features ├──1:*── entitlement_overrides (override: grandfather / ดีลพิเศษ) ├──1:*── usage_counters (metered ต่อ period) │ └──1:*── projects · machine_profiles … (domain · org-scoped + RLS)
plans
แพ็กที่ลูกค้าซื้อ
  • code (free/plus/…)
  • price_cents · interval
plan_entitlements
ตาราง tier × feature (matrix ในรูป DB)
  • bool_value
  • limit_value (null=∞)
features
แคตตาล็อกฟีเจอร์ที่ gate ได้
  • key · category
  • kind · unit
subscriptions
org ปัจจุบันอยู่แพ็กไหน
  • org_id → plan_code
  • status · period_end
entitlement_overrides
สิทธิ์เฉพาะ org (มี expires_at)
  • bool/limit override
  • reason · expires_at
usage_counters
การใช้ metered ต่อรอบ
  • period (YYYY-MM)
  • used_value
Resolver: has_feature() · feature_limit() · can_consume() · consume() · assert_feature() — ลำดับ resolve: override (ยังไม่หมดอายุ) → plan default → ไม่มีสิทธิ์ ทั้งหมดเป็น security-definer เพื่ออ่านตาราง billing ข้าม RLS ได้อย่างปลอดภัย
