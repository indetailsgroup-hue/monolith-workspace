UX Tenet Audit — ทั้งระบบ (v1)

"หน้าบ้านง่ายจนคนรัก หลังบ้านรับความยากไว้เอง" — ตรวจทุก persona/flow

ตรวจทุก persona จาก PRD §4 (ฉบับ reconcile กับ 38 ตำแหน่ง) ด้วยไม้บรรทัด Governing UX Tenet (PRD §1): ต่อ 1 งานวัด (a) จำนวน tap/การกระทำ (b) ศัพท์ระบบที่ผู้ใช้เห็น — คู่กับ audit เฉพาะทางของงานติดตั้งที่ .kiro/specs/installation-pm/ux-tenet-audit-v1.md ข้อยกเว้นที่ประกาศชัด: ผู้ใช้ "มืออาชีพของระบบ" (Designer/Draftsman/Admin/DevOps) เครื่องมือซับซ้อนได้เพราะความซับซ้อนคือเนื้องานเขา — tenet เข้มสุดกับ ผู้ใช้ปลายทาง/หน้างาน/นาน ๆ ใช้ที (ช่าง ลูกค้า ช่างวัด โรงงาน ผู้อนุมัติ) คะแนน: 🟢 ผ่าน · 🟡 เสี่ยง (ดันเข้าหลังบ้านได้) · 🔴 leak (ผลักความยากให้ผู้ใช้ผิดที่)

1. Customer (ลูกค้า) — เข้มสุดเพราะเป็นคนจ่ายเงินและไม่มีทางถูกอบรม

Flow ตอนนี้ คะแนน หมายเหตุ/เฝ้าระวัง
ดูแบบ 3D ลิงก์จาก LINE เปิดดูเลย ไม่ต้องสมัคร account ✅ 🟢 เฝ้าระวัง: ลิงก์ต้องไม่หมดอายุเร็วจนลูกค้ากดแล้วเจอ error — ถ้าหมดอายุให้ขอลิงก์ใหม่ได้ใน 1 tap
อนุมัติแบบ/งาน Flex การ์ดเดียว กดปุ่มเดียว + web fallback ✅ 🟢 เฝ้าระวัง: ภาษาบนการ์ด/ปุ่มต้องเป็นภาษาคน 100% ("อนุมัติแบบห้องครัว" ไม่ใช่ "Approve Released_Spec v3") — เพิ่มเป็นเกณฑ์ review template
รับความคืบหน้า curated update ในกลุ่ม (มนุษย์เลือกส่ง) 🟢 ตามมติ owner แล้ว
แก้แบบ/ปฏิเสธ กดปฏิเสธ → ? 🟡 ต้องมีช่องบอกเหตุผลแบบง่าย (ปุ่มเหตุผลยอดฮิต + พิมพ์เอง) แล้วระบบ route กลับเข้า revision workflow เอง — ลูกค้าไม่ควรต้องรู้ว่า "ต้องติดต่อใครยังไงต่อ"

2. Surveyor (ช่างวัดพื้นที่) — ทุกวัน, หน้างาน

Flow ตอนนี้ คะแนน ดันเข้าหลังบ้าน
รับงานวัดวันนี้ (spec ยังไม่ระบุ) 🟡 ให้เหมือนช่างติดตั้ง: "งานวัดของฉันวันนี้" — ที่อยู่ + แผนที่ + ข้อมูลจาก sale แนบครบ (PFMEA: ลืมปริ้นเอกสาร = ความเสี่ยง sev 8 → ระบบแนบให้เอง ไม่ต้องปริ้น)
บันทึกผลวัด form site_survey (SiteSurveyZone มี schema แล้ว) 🟡 ฟอร์มต้องไล่ตามลำดับที่ช่างวัดจริง (เอียงพื้น→ระยะ→ช่องเปิด→ไฟ) + รูปแนบต่อ zone + ใช้ได้บนมือถือถนัดมือเดียว; ห้ามให้เลือก project (มาจากงานที่กดเข้า)
แบบฟอร์มความต้องการ (wardrobe นับเสื้อผ้า) กระดาษเดิม 🟡 แปลงเป็น checklist ติ๊ก+ตัวเลข ใน PWA — โครงตามใบจริง ข้อมูลออกวัดพื้นที่ฯ (LADY/MAN) ที่ช่างคุ้นอยู่แล้ว

3. Factory Operator (โรงงาน) — ทุกวัน

Flow ตอนนี้ คะแนน ดันเข้าหลังบ้าน
รับงาน/ดาวน์โหลด bundle Factory Server (LAN) เลือกงาน → โหลด ZIP ตรง dialect เครื่อง 🟢 ดีอยู่: เลือกเครื่องครั้งแรกครั้งเดียว ระบบจำ dialect
ตรวจ receipt CLI monolith-receipt-verify 🔴 → 🟡 CLI สำหรับ operator = leak ชัด ๆ — Factory Server UI ต้อง verify ให้อัตโนมัติทุก bundle แล้วโชว์ ✅/❌ ใหญ่ ๆ ก่อนปุ่มดาวน์โหลด (operator ไม่ต้องรู้ว่ามีการ verify); CLI คงไว้สำหรับ auditor/offline dispute (Golden Matrix P13.4) — เป็น power tool ไม่ใช่ flow ปกติ
แจ้งของเสีย/เครื่องมีปัญหา (ไม่มี flow) 🟡 ให้เหมือนช่างติดตั้ง: ถ่ายรูปที่สถานี → ปุ่ม "แจ้งปัญหา" → route ไปซ่อมบำรุง/QA (pattern #ปัญหา เดียวกัน — reuse)

4. Office Staff / ผู้อนุมัติ (ขาย/วัด/วางแผน) — ทุกวัน

Flow ตอนนี้ คะแนน ดันเข้าหลังบ้าน
อนุมัติ work item LINE คลิกเดียว + web fallback ✅ 🟢 pattern ถูกแล้ว
เห็นสถานะงาน 8 ขั้น workflow console 🟡 ภาษาบนจอ = ชื่อขั้นภาษาคน ("รอวัดพื้นที่" ไม่ใช่ "AREA_MEASUREMENT_PENDING"); การส่งต่อขั้น = ปุ่มเดียว ระบบตรวจเงื่อนไขเองแล้วบอกว่าขาดอะไรเป็นภาษาคน
Sales เก็บ customer_requirement (16 fields) ฟอร์ม 🟡 16 ช่องคือเยอะสำหรับคุยกับลูกค้าไปกรอกไป — จัดเป็น wizard ตามบทสนทนาจริง (ใคร→ที่ไหน→ทำอะไร→สไตล์→วัสดุ→งบ) + ค่า TBD ได้ทุกช่องที่ไม่ critical + ดึงจากแชท LINE อัตโนมัติ เมื่อลูกค้าพิมพ์มาแล้ว (อย่าให้ sale พิมพ์ซ้ำสิ่งที่ลูกค้าเขียนเอง)

5. Executive Owner — รายสัปดาห์

Escalation (RPN/งบเกิน) เป็นการ์ดถึงมือทันที ✅ · ตัดสินใจ = ตอบการ์ด/สั่งใน Copilot — 🟢 · เฝ้าระวัง: การ์ด escalation ต้องสรุป "ต้องตัดสินอะไร + ผลของแต่ละทาง" ในการ์ดเดียว ไม่ใช่ลิงก์ไปอ่านเอกสารยาว

6. Accountant — หลักต่างออกไป (rigor คือเนื้องานเขา)

บัญชีคือคนที่รับความยากแทนคนอื่นทั้งระบบ — tenet สำหรับเขาไม่ใช่ "ซ่อน rigor" แต่คือ "อย่าให้ทำงานซ้ำซ้อน":

  • ✅ ทำถูกแล้ว: OCR กรอกให้ก่อน + มนุษย์ยืนยัน (ไม่ใช่พิมพ์เอง), fraud flags ชี้เป้า (ไม่ใช่ไล่หาเอง), verify queue รวมที่เดียว
  • 🟡 เกณฑ์เพิ่ม: หน้า verify ต้องเห็น รูปเอกสาร + ค่าที่ OCR อ่าน เคียงข้างกัน แก้เฉพาะช่องที่ผิด (corrected_fields — มีใน 0078 แล้ว ✅) + ปุ่มยืนยันเดียว; ห้าม re-type ทั้งใบ

7. Designer / Draftsman / 3D (มืออาชีพของระบบ — ข้อยกเว้น)

เครื่องมือ CAD/CAM ซับซ้อนได้ (เนื้องาน) — จุดเดียวที่ tenet ยังบังคับ:

  • 🟡 ภาษา blocker ของ Gate: "เจาะชิดขอบเกิน — กด Focus เพื่อดูจุด, กด Fix เพื่อแก้อัตโนมัติ" ✅ ดีแล้ว แต่ต้องคุมทุก G-code ใหม่ (รวม G12 curved) ให้มี (ก) ภาษาคนบอกว่าผิดอะไร (ข) ปุ่มพาไปดู (ค) ปุ่มแก้อัตโนมัติเมื่อมี — ห้ามโชว์แค่รหัส error

กติกาถาวรที่ตกผลึกจาก audit ระบบ (เพิ่มจาก D-12 ของ installation-pm)

# กติกา ใช้กับ
S1 Verify อัตโนมัติ ผู้ใช้เห็นแค่ ✅/❌ — CLI/เครื่องมือเทคนิคเป็นของ auditor ไม่ใช่ flow ปกติ Factory receipt, ทุก integrity check
S2 ภาษาคนทุกจอผู้ใช้ปลายทาง — ชื่อขั้น/ปุ่ม/error ห้ามเป็นรหัส/ENUM/ศัพท์ spec workflow console, gate blockers, Flex ทุกใบ
S3 อย่าให้ใครพิมพ์สิ่งที่ระบบรู้แล้ว — OCR ก่อนมนุษย์แก้, ดึงจากแชทก่อน sale กรอก, infer ก่อนถาม capture ทุก type, customer_requirement
S4 ปฏิเสธ/ผิดพลาดต้องมีทางไปต่อใน 1 tap — reject มีช่องเหตุผลง่าย + ระบบ route เอง, ลิงก์หมดอายุขอใหม่ได้ทันที customer approval, ทุก error state
S5 ทุกคนหน้างานมีปุ่ม "แจ้งปัญหา" pattern เดียวกัน (รูป → ปุ่ม → route เอง) ช่างติดตั้ง ✅, โรงงาน, ช่างวัด
S6 คนรับ rigor (บัญชี/QA) = ห้ามทำซ้ำซ้อน — ระบบเตรียมให้แก้เฉพาะจุด ไม่ re-type verify queues ทุกอัน

รายการแก้ที่เข้าคิว (จะผูกเข้า spec/tasks ที่เกี่ยวข้อง)

  1. Factory Server: auto-verify receipt + ✅/❌ บน UI (S1) → เพิ่มเข้า factory server backlog
  2. Gate blockers: เกณฑ์ (ภาษาคน + Focus + Fix) เป็นข้อบังคับของ G-code ใหม่ทุกตัว รวม G12 curved → note เข้า curved-panel-system spec
  3. customer_requirement wizard + ดึงจากแชท (S3) → note เข้า field-purchase/installation-pm sibling (capture UX)
  4. Survey mobile form ตามลำดับวัดจริง + งานวัดของฉันวันนี้ → เข้า installation-pm/measure backlog
  5. Reject flow ลูกค้า + ลิงก์หมดอายุ (S4) → เข้า template governance checklist
  6. ปุ่มแจ้งปัญหาที่โรงงาน (S5) → reuse pattern จาก installation-pm

Audit นี้ประเมินจาก spec/PRD ปัจจุบัน — flow ที่ยังไม่ implement ให้ถือกติกา S1–S6 เป็นข้อบังคับตอน implement; flow ที่ implement แล้ว (LINE approve, capture verify) ให้ตรวจกับของจริงอีกครั้งตอน pilot Wave 1–2 แล้วอัปเดตคะแนน
