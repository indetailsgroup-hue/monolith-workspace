# รายงานประเมินความเข้ากันได้ (Compatibility Assessment) — KDT KD-612NS

> `assessment: MCA-KD-612NS@0.1` · `profile: kdt.six_sided_drill.kd_612ns@0.1-research-draft` · `contract: DELIVERY-KD-612NS@0.1-draft`
> [EN edition](compatibility-assessment.en.md)

> **การปล่อยงานผลิต: PROHIBITED** — สถานะการประเมิน: `NOT_ASSESSED`

**การปล่อยงานผลิต: PROHIBITED (ห้าม)** — สถานะการประเมินยังเป็น `NOT_ASSESSED` ทุกค่าในคลังนี้มาจากเอกสารสาธารณะเท่านั้น (**Verified in documents**) ไม่เคยมาจากการเปิดใช้งานเครื่องจริง ห้ามปล่อยงานสู่การผลิตจนกว่าจะผ่าน gate การยืนยันหน้าเครื่องครบทุกข้อด้านล่าง

## A. สถานะการประเมิน

- สถานะการประเมิน: **NOT_ASSESSED**
- การปล่อยงานผลิต: **PROHIBITED**

## B. ขอบเขตของการประเมินนี้

นี่คือ **การตรวจความสามารถจากเอกสารเท่านั้น** โดยเทียบหลักฐานแค็ตตาล็อกสาธารณะกับความคาดหวังของแพลตฟอร์ม MONOLITH และแสดงข้อขัดแย้ง/อุปสรรคที่ยังไม่คลี่คลาย **ไม่มี** การยืนยันการทำงานจริง และ **ไม่ให้** อำนาจการผลิตใด ๆ

## C. กฎการตรวจรูปทรง / ขอบเขตชิ้นงาน

- ตรวจความยาว/กว้าง/หนาของแผ่นกับ envelope ที่มีเอกสารก่อนเจาะทุกครั้ง
- ยืนยันพิกัดรูอยู่ในเขตเจาะที่เอื้อมถึงทั้งหกด้าน
- ปฏิเสธงานที่ขนาดแผ่นเกินค่า min/max ที่ระบุใน envelope

## D. คอนโทรลเลอร์ / ฟอร์แมต / ทูล / จุดกำเนิด / การส่งไฟล์จริง

สิ่งเหล่านี้เป็นระดับ **เครื่องจริง** และยังเป็น **Unknown** เว้นแต่มีหลักฐานของรุ่นนี้โดยตรง ดูโปรไฟล์คอนโทรลเลอร์และสัญญาส่งไฟล์ในหน้าโปรไฟล์เครื่อง

## E. ข้อขัดแย้งและอุปสรรคที่ยังไม่คลี่คลาย

**ช่องว่างหลักฐานที่บันทึกไว้:**
- `controller_hmi`

- `automation`

## F. ลำดับการตรวจสอบที่บังคับ

1. **งานตัวอย่างที่รู้ว่าดี (immutable known-good job)** — เริ่มจากงานที่เครื่องรันถูกต้องอยู่แล้ว เก็บไฟล์ต้นฉบับแบบไม่แก้ไข
2. **ตรวจ parser / import** — ยืนยันว่างาน parse และ import ได้บนคอนโทรลเลอร์/หน้า import ของเครื่องเอง
3. **จำลอง (ถ้ามี)** — รันการจำลองบนคอนโทรลเลอร์หรือ CAM ที่เครื่อง/ซอฟต์แวร์รองรับ
4. **Dry-run / air-cut หรือวิธีปลอดภัยเทียบเท่า** — รัน dry-run, air-cut หรือวิธีปลอดภัยของตระกูลนั้น ก่อนตัดวัสดุจริง
5. **ตรวจชิ้นงานแรก (first-article)** — วัดชิ้นงานจริงชิ้นแรกเทียบกับพิกัดความเผื่อของแบบ
6. **มนุษย์รับงาน (human acceptance)** — ผู้เชี่ยวชาญยืนยันรับผลอย่างชัดเจน ไม่มีการปล่อยเครื่องอัตโนมัติ

## G. ขอบเขต Interoperability & Validation

OCCT / PythonOCC ใช้เพื่อ **Interoperability & Validation เท่านั้น** ไม่ใช่ **Manufacturing Authority** ไฟล์ต้นฉบับ **ห้ามแก้ไข (immutable)** การแปลงหรือ heal เรขาคณิตใด ๆ จะได้ **Versioned Derived Artifact** ซึ่งต้องผ่านการรับงานโดยมนุษย์อีกครั้ง และ **ไม่มีการปล่อยเครื่องอัตโนมัติ**
