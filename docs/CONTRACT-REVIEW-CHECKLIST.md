# Checklist ตรวจเอกสารผูกพันก่อนใช้กับลูกค้าจริง (legal gate)

> **ที่มา:** ปรับจาก C8 Agreement Review Checklist + Caveat Library ของชุด Interior Design Studio System
> เข้าบริบท DAPH ตามมติ **ADR-044 R-6** — ครอบคลุม **สัญญา (0117)** และ **Variation Order (0132)**
>
> **กติกาสำคัญ: ติ๊กครบ ≠ legal approval** — เอกสารนี้เป็นเครื่องมือ*เตรียมชุดส่งให้ทนายตรวจอย่างเป็นระบบ*
> ไม่ใช่คำแนะนำทางกฎหมาย ภาษี บัญชี วิศวกรรม หรือประกันภัย

## เมื่อไหร่ต้องผ่าน gate นี้

- [ ] ก่อนใช้ **สัญญา** ฉบับ generate (skeleton ใน `rpc_field_generate_contract`) กับลูกค้าจริงครั้งแรก
- [ ] ก่อนใช้ **Variation Order** ฉบับ generate (`rpc_field_create_variation`) กับลูกค้าจริงครั้งแรก
- [ ] ทุกครั้งที่แก้ skeleton ของเอกสารทั้งสอง (แก้ = ต้อง review รอบใหม่เฉพาะวรรคที่แก้)
- อ้างใน runbook: [OPS-RUNBOOK-Provision.md](OPS-RUNBOOK-Provision.md) — ก่อน V4/V10

|---|---|---|---|
| 1 | คู่สัญญา + อำนาจลงนาม | ใครลงนามฝั่ง DAPH/ลูกค้า มีอำนาจตามกฎหมายหรือไม่ | โครงบริษัท + ตัวอย่างสัญญา generate |
| 2 | Scope / ข้อยกเว้น / งานแก้ | ขอบเขต+สิ่งที่ไม่รวม ชัดพอสำหรับ dispute หรือไม่ · จำนวนรอบแก้แบบ | ใบ requirement + body สัญญา + ADR-037 (requote) |
| 3 | เงิน: งวด 50/30/15/5 / จ่ายช้า / ค่าใช้จ่ายเพิ่ม | เงื่อนไขงวด+ผลของการไม่จ่าย (เราหยุดผลิตได้เมื่อไหร่ — soft gate 0108) ถูกกฎหมายหรือไม่ | แผนงวดจริง + fn_payment_fire flow |
| 4 | การเปลี่ยนแปลงงาน | วรรค VO ("งานส่วนที่เปลี่ยนเริ่มหลังลงนาม") ผูกพันจริงหรือไม่ · ราคา VO โต้แย้งได้แค่ไหน | ตัวอย่าง VO-001 generate |

| 6 | ความรับผิด / ประกัน 1 ปี + ฟิตติ้งตามแบรนด์ | วรรคประกันสองชั้น (AS-2.1) เขียนถูกต้อง จำกัดความรับผิดพอหรือไม่ | ADR-042 + การ์ดประกัน |
| 7 | ทรัพย์สินทางปัญญา / รูปงาน / รีวิว | DAPH ใช้รูปบ้านลูกค้าทำ portfolio/การตลาดได้แค่ไหน ต้อง consent แบบไหน | flow รูปในระบบ + การ์ดรีวิว (AS-3) |
| 8 | PDPA | ข้อมูล+รูปในกลุ่ม LINE / ระบบ — วรรค consent ครบหรือไม่ | นโยบาย PDPA ระบบ (เก็บเฉพาะรูป/#ปัญหา/member events) |
| 9 | ภาษี: **หัก ณ ที่จ่าย / VAT บนใบเสร็จ-ใบแจ้งงวด** | wording ใบเสร็จ RC + ใบแจ้งงวดถูกต้องตามประกาศสรรพากรหรือไม่ (Localization Matrix C8) | ตัวอย่างใบเสร็จ + ใบแจ้งงวด generate |
| 10 | PDPA เพิ่มเติม | รูปบ้านลูกค้า/testimonial ใน portfolio — consent แบบไหน | flow รูป + การ์ดรีวิว |
| 11 | ระงับข้อพิพาท / กฎหมายที่ใช้ | ศาล/อนุญาโตฯ เหมาะกับขนาดคดีของเราหรือไม่ | — |
| 12 | **ล็อกสเปกถึงรหัสสินค้า** | โครง (เช่น HMR 15mm ยี่ห้อ+รหัส) / ฟิตติ้ง (ยี่ห้อ+รุ่น) / ลามิเนต (ยี่ห้อ+รหัสสี) ระบุในสัญญา+Material Board — สเปกคลุมเครือ = ช่องหมกเม็ด (C18/ADR-052) | cabinet & wall list 7 fields + BOM |

## Clauses เฉพาะ Variation Order

- [ ] วรรค "ไม่เริ่มงานส่วนที่เปลี่ยนจนกว่าลงนาม" — ยกเว้นกรณีเร่งด่วน (written urgent instruction) ต้องมีไหม
- [ ] ผลกระทบราคา ± ผูกกับการปรับงวดอย่างไร (ระบบ: F3 ปรับแผนผ่านเส้น 0108 — ทนายตรวจว่าวรรครองรับ)
- [ ] hidden condition (สภาพหน้างานซ่อน) — ภาระพิสูจน์อยู่ฝั่งไหน

## Pass standard (ก่อนปิด gate)

- [ ] เก็บ comment ทนายครบทุกวรรค + assign การแก้ให้เจ้าของงาน
- [ ] ไม่มี open legal question ซ่อนอยู่ในฉบับ final
- [ ] แก้ skeleton ในโค้ด (`0117` / `0132`) ตาม comment แล้ว migration ใหม่ + ทดสอบ
- [ ] ลบ marker "โครงร่าง v1 — รอทนาย review skeleton" ออกจาก body เมื่อผ่านแล้วเท่านั้น
- [ ] จดวันที่+ชื่อทนาย+ฉบับที่ review ลง audit (`installation_audit_log` event `legal_review_passed` — insert มือครั้งเดียว)

## Caveat มาตรฐาน (จาก Caveat Library — ใช้กับเอกสารที่ยังไม่ผ่าน gate)

> "เอกสารนี้เป็นแบบเอกสารเพื่อการดำเนินงาน (business template) ยังไม่ผ่านการตรวจโดยที่ปรึกษากฎหมาย
> ก่อนใช้ลงนามจริงต้องผ่านการตรวจตามเขตอำนาจกฎหมายและสัญญาจริงของบริษัท"
