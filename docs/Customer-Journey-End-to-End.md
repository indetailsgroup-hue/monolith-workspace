# เส้นทางลูกค้า DAPH — ตั้งแต่เห็นโฆษณา จนส่งมอบบ้าน (ฉบับเจาะลึกทุกรายละเอียด)

> อิงระบบที่สร้างเสร็จจริง ณ 7 ก.ค. 2026 (migrations 0001–0100 · workflow 134/134 · ADR-034…040)
> ทุกก้าวระบุ: ลูกค้าเห็นอะไร · ทีมทำอะไร · ระบบทำอะไรหลังบ้าน · อะไรกันความผิดพลาดไว้
> หลักออกแบบทั้งเส้น (PRD §1): **หน้าบ้านง่ายจนทุกคนรัก — หลังบ้านรับความยากไว้เอง**

## 0. แผนที่การเดินทาง (ภาพรวมก่อนเจาะ)

```
เห็นโฆษณา/เว็บ → add LINE OA → คุยกับ Sale → ใบบันทึกความต้องการ → เปิดงานในระบบ
      │
      └→ [8 ขั้น canonical]
         Sale → วัดพื้นที่ → ออกแบบ(G1) → 3D นำเสนอ(G2) → วางแผนผลิต(G4) → 3D Final(G3) → ผลิต → ติดตั้ง
                              ▲ ลูกค้าเซ็น    ▲ ลูกค้าเซ็น                     ▲ ลูกค้าเซ็น
      ┌──────────────────────────────────────────────────────────────────────┘
      └→ เปิด "บ้าน" ในระบบติดตั้ง → กลุ่ม LINE 2 กลุ่ม → T0 → ติดตั้งรายห้อง/รายเลน
         → ใบปิดบ้าน (หัวหน้าทีมเท่านั้น) → การ์ดตรวจรับใน LINE → ลูกค้ากดรับงาน → ปิดโปรเจกต์ 🏠
```

| ช่วง | ลูกค้าเห็น | สถานะในระบบ |
|------|-----------|--------------|
| 1. รู้จัก | โฆษณา/เว็บ → ปุ่ม add LINE | conversation + ตัวตนลูกค้าเกิดอัตโนมัติ |
| 2. คุย/Qualify | แชทกับ Sale ตามปกติ | ใบบันทึกความต้องการ → verify → เปิด work item |
| 3. ออกแบบ | การ์ดขออนุมัติแบบ 3 ครั้ง (G1/G2/G3) | approval loop + design lock ต่อ gate |
| 4. ผลิต | (เงียบ — ทีมทำงาน) | Factory Packet + ใบงานสถานี |
| 5. ติดตั้ง | อยู่กลุ่ม LINE บ้านตัวเอง เห็นเฉพาะ curated update | บ้าน→ห้อง→เลนช่าง + รูปหลักฐานทุกเลน |
| 6. ส่งมอบ | การ์ดตรวจรับ กดปุ่มเดียว | project completed + audit ครบทั้งเส้น |

## 1. รู้จัก → add LINE OA (นาทีที่ 0 ของความสัมพันธ์)

**ลูกค้าเห็น:** โฆษณา/เว็บ → กด add เพื่อน → ได้ข้อความต้อนรับ (Greeting Message — ตั้งใน LINE console, runbook C3.6)

**ระบบทำทันทีที่ add/ทักครั้งแรก** (webhook ผ่านการตรวจลายเซ็น HMAC ทุกครั้ง — ปลอมไม่ได้):

1. สร้าง **conversation** — แชทมีตัวตนใน DB, lead ไม่มีวันหายเงียบ
2. สร้าง **ตัวตนลูกค้า** (`line_oa_customer_identity`) — **ตัวตนเดียวนี้จะถูกใช้ยาวไปจนถึงวันตรวจรับบ้าน**: เป็นคนเดียวกันที่เซ็นแบบ G1/G2/G3 และกดรับงานวันสุดท้าย
3. ทุก event มี idempotency (`webhook_event_id` UNIQUE) — LINE ส่งซ้ำกี่รอบระบบก็ไม่เพี้ยน

**สิ่งที่ระบบ*ไม่*ทำ (ตั้งใจ):** ไม่มี bot ตอบแชทอัตโนมัติแบบ free-text (กติกา template-only — ทุกข้อความขาออกเป็นแบบที่ผ่านการอนุมัติแล้วเท่านั้น) · Sale คุยขายผ่านแอป LINE OA Manager ตามปกติ ไม่ต้องเรียนรู้อะไรใหม่

## 2. Qualify → ใบบันทึกความต้องการ (ประตูทางเข้าของทุกอย่าง)

**ลูกค้าเห็น:** แค่คุยกับ Sale — คำถามที่ Sale ถามคือแบบสอบถามมาตรฐานที่ทำร่วมกับฝ่ายดีไซน์

**Sale ทำ:** กรอก **ใบบันทึกความต้องการ** (`customer_requirement` — ฟอร์มมากับ PWA Wave A) ครบ 3 หมวด:

| หมวด | ฟิลด์ | ทำไมพลาดไม่ได้ |
|------|-------|----------------|
| ติดต่อ | ชื่อ, เบอร์, email, LINE ID, ที่อยู่ | PFMEA Sale ขั้น 1: เก็บไม่ครบ/ผิด = **"Scrap 100%"** ทั้ง downstream |
| ขอบเขต (9 ฟิลด์จาก Qualify) | โครงการ, Type บ้าน, ตรม., ห้องที่ทำ, สไตล์, วัสดุโครงสร้าง/โครงตู้/ปิดผิว, แบรนด์ fitting | เสนอราคาผิดเรทถ้าวัสดุผิด (PFMEA ขั้น 2) — ค่าที่ลูกค้ายังไม่ตัดสินใจต้องกรอก "TBD" ห้ามปล่อยว่าง |
| ดีไซน์ | Mood & Tone, Function | ดีไซน์เนอร์เริ่มงานผิดทางถ้าไม่มี |

**ระบบทำ (capture spine — เส้นเดียวกับเอกสารทุกชนิดของบริษัท):**

```
กรอก (ingest, idempotent) → verify (มนุษย์ยืนยัน critical fields ตามกฎ PFMEA) → approve
  → promote = ระบบเปิด work_item ขั้น Sale ให้อัตโนมัติ (adapter 'work_item_open' — 0092)
```

รายละเอียดที่คิดไว้แล้วในจุด promote:

- **ผูกตัวตนลูกค้า** (0100): ฟอร์มส่ง `customer_id` (ตัวตนจากข้อ 1) → work item ได้ `primary_customer_id` → **นี่คือเหตุผลที่การ์ดอนุมัติแบบ G1/G2/G3 จะถูกส่งถึงลูกค้าคนนี้จริง** · ลูกค้า walk-in ไม่มี LINE → ระบบ degrade เป็นอนุมัติภายใน (ดีไซน์รองรับ) + audit บอกชัดทุกใบว่าผูกหรือไม่
- **PDPA**: ข้อมูลบุคคล (ชื่อ/เบอร์/ที่อยู่) อยู่ที่ใบ capture ซึ่ง RLS คุมแน่น — *ไม่ copy* ลง work item
- Fail-safe ที่จุด commit: ไม่มีชื่อ/ช่องทางติดต่อ → ระบบปฏิเสธการเปิดงาน (กันข้อมูลขยะเข้า pipeline)
- Idempotent: promote ซ้ำ = งานใบเดิม ไม่เปิดซ้ำ

## 3. แปดขั้น canonical — เครื่องจักรที่พางานเดิน

งานทุกใบเดินตาม **canonical process 8 ขั้น** โดยแต่ละขั้นมีเจ้าของและผู้อนุมัติจาก **RACI Master Matrix** (knowledge import ที่ approve แล้วเท่านั้น — ไม่มี RACI = ระบบ block ไม่เดา):

| # | ขั้น | ทีม | ลูกค้าเกี่ยวอย่างไร |
|---|------|-----|---------------------|
| 1 | Sale | ฝ่ายขาย | คุย/ตกลงเบื้องต้น |
| 2 | Area Measurement | ทีมวัดพื้นที่ | นัดเข้าวัดหน้างาน — ข้อมูลวัด (ขนาด/MEP/รูป) เข้าระบบเป็น SiteSurveyZone มี version |
| 3 | Designer | ดีไซน์ | **เซ็นอนุมัติ G1** — Mood & Tone / สไตล์ / สี |
| 4 | 3D_Presentation | ทีม 3D | **เซ็นอนุมัติ G2** — layout การจัดวางเฟอร์นิเจอร์ |
| 5 | Production Planning | วางแผนผลิต | (ภายใน — G4: ปล่อยงานเข้าผลิต; RPN สูง/งบเกินเพดาน → ยกระดับถึงผู้บริหาร) |
| 6 | 3D_Rendering_Final | ทีม 3D | **เซ็นอนุมัติ G3** — วัสดุ/สีจริงขั้นสุดท้าย |
| 7 | Factory | โรงงาน | (ภายใน — Factory Packet + ใบงานสถานี + FTR checklist) |
| 8 | Installation | ทีมติดตั้ง | ช่วงที่ลูกค้าใกล้ชิดระบบที่สุด — ดูข้อ 5 |

### กลไกอนุมัติ (ทำงานเหมือนกันทุกขั้น — สร้างครั้งเดียวใช้ทั้งบริษัท)

- งานมาถึงขั้นที่ต้องอนุมัติ → ระบบ **resolve ผู้อนุมัติจาก RACI** (คน + ลูกค้า ถ้าเป็น G1/G2/G3) → quorum = **unanimous** (ต้องครบทุกเสียง รวมลูกค้า)
- **ลูกค้าเซ็นผ่าน LINE** (Flex การ์ด) หรือ **web fallback** (ลิงก์อายุสั้น) — กดปุ่มเดียว ไม่ต้อง login อะไร
- มีคนไม่ว่าง? — ระบบ delegation: ผู้อนุมัติมอบอำนาจล่วงหน้าได้ (จำกัดขั้น/ช่วงเวลา/site) — audit ครบ
- **นาฬิกา SLA เดินทุกใบ**: เตือนที่ 50%/100% (push ส่วนตัว) → เกินเวลา = escalate อัตโนมัติ (ลูกค้าเงียบ → แจ้ง PM ตามหา)
- **Quiet Hours 20:00–08:00**: แจ้งเตือนทีม (ไม่เร่งด่วน) ไม่รบกวนกลางคืน — เก็บรวมเป็น **สรุปเช้า 08:00** ฉบับเดียว; เฉพาะงานความรับผิดชอบส่วนตัวเท่านั้นที่ข้ามได้

### ทุกครั้งที่ gate ผ่าน = ระบบ "ล็อกแบบ"

G1 ผ่าน → ล็อกชุด Mood&Tone · G2 → ล็อก layout · G3 → ล็อกวัสดุ · G4 → ล็อกแบบก่อสร้าง — สิ่งที่ลูกค้าเซ็นแล้วจะไม่ถูกแก้เงียบ ๆ (ปลดล็อกได้ทางเดียว: scope change ที่อนุมัติแล้ว — ดูข้อ 4)

## 4. ถ้าลูกค้าขอแก้แบบ (เกิดขึ้นจริงเสมอ — ระบบออกแบบรอไว้แล้ว)

ตัวอักษร G บอก**ต้นทุนของการแก้** (G1 แพงสุด — แก้ Mood&Tone = รื้อทุกอย่างที่ตามมา):

```
ลูกค้ากด "ขอแก้ไข" ที่ gate ใด ๆ → ระบบ classify อัตโนมัติ:
  • แก้ในสิ่งที่ยังไม่เคยเซ็น            → งานแก้ปกติ (ฟรี — ยังอยู่ในรอบออกแบบ)
  • แก้สิ่งที่เซ็นล็อกไปแล้ว (scope change) → เข้าเส้น re-quote:
      งานหยุดรอ → PM+ผู้บริหารสรุปราคาใหม่ → ลูกค้ารับราคา
      → ครบทั้งคู่: ปลดล็อก gate นั้น + งานถอยกลับไปขั้นของ gate (ADR-037)
      → เดินวงจรอนุมัติใหม่ตามปกติ → เซ็นใหม่ → ล็อกใหม่อัตโนมัติ
  • ความบกพร่องของ DAPH                → แก้ฟรี + บันทึกเป็น defect
```

จุดสำคัญ: **แบบที่แพงขึ้นถูกเซ็นซ้ำโดยผู้มีอำนาจจริงเสมอ** — ไม่มีการปลดล็อกเงียบ ๆ และเกินโควตาแก้ฟรี ระบบเตือนอัตโนมัติว่าเข้าเขตคิดเงิน

## 5. ช่วงติดตั้ง — บ้านจริง ทีมจริง (เจาะลึกสุดเพราะลูกค้าใกล้ระบบที่สุด)

### 5.1 เปิด "บ้าน" ในระบบ

Office เปิดโปรเจกต์บ้าน (ผูกกับ work item ขั้น Installation) พร้อม **preset บ้านมาตรฐาน 5 ห้อง** (ครัว + นั่งเล่น + นอนใหญ่ + นอน ×2 — ปรับตามบ้านจริง) → ระบบสร้างโครงให้ทันที:

```
บ้าน 1 หลัง (หัวหน้าทีม 1 คน = ผู้อนุมัติเริ่ม/จบงาน)
 ├─ ห้อง ×5 (ชื่อภาษาคน เช่น "ห้องนอนใหญ่ ชั้น 2")
 │   └─ เลนช่าง ×3 ต่อห้อง = "ช่างคนที่ 1/2/3" ทำขนานกัน (checklist คนละชุดจาก template จริง
 │       ที่ใช้มา 5 ปี — Installation.xlsx; งานประปาเป็นรายการ conditional ช่างทุกคนทำได้)
 └─ ทีมรวม: 3 ช่าง × 5 ห้อง + หัวหน้า 1 = 16 คน
```

### 5.2 กลุ่ม LINE 2 กลุ่ม — โลกของทีม กับโลกของลูกค้า แยกกันที่ระดับ DB

| | กลุ่มทีม (internal) | **กลุ่มลูกค้า (customer)** |
|--|--------------------|---------------------------|
| สมาชิก | sale + ทีมหมุนตามเฟส (วัด→ออกแบบ→ติดตั้ง) | **ลูกค้า + sale + หัวหน้างาน** (ช่างไม่อยู่) |
| bot ทำอะไร | รับรูปเข้าระบบ, รับ `#ปัญหา`, แจ้งเตือนงาน | ส่งเฉพาะ**ความคืบหน้าที่มนุษย์คัดแล้ว** + นัดหมาย + การ์ดอนุมัติ/ตรวจรับ |
| กันข้อมูลหลุด | — | **Guardrail ที่ DB**: ข้อความเข้ากลุ่มลูกค้าได้เฉพาะ template ที่ประกาศ `audience=customer` — ต้นทุน/เรื่องภายในส่งผิดกลุ่ม = ระบบโยน error ไม่ใช่แค่เตือน; รูปกลุ่มทีม**ไม่มีวัน forward อัตโนมัติ** |

การผูกกลุ่มกับบ้าน: office ออก**รหัสผูก** (หมดอายุ 48 ชม. ใช้ได้ 2 ครั้ง) → หัวหน้า/sale ตั้งกลุ่ม + เชิญ bot → พิมพ์ `#ผูก <รหัส> ทีม|ลูกค้า` → **bot ตอบชื่อบ้านกลับมาในกลุ่ม** ("เชื่อมกลุ่มนี้กับ 'บ้านคุณสมชาย รามอินทรา' แล้วครับ ✅") — ผูกผิดหลังเห็นกันทั้งกลุ่มทันที ก่อนข้อมูลใดไหล (0100) · ผู้ผูกต้องเป็นพนักงานที่ผูกตัวตนแล้วเท่านั้น · **การอยู่ในกลุ่ม ≠ สิทธิ์ในระบบ** — สิทธิ์ข้อมูลมาจาก membership + RLS เสมอ

### 5.3 เริ่มงาน: T0 → หัวหน้าทีมอนุมัติ

ก่อนเริ่ม หัวหน้าทีมติ๊ก **T0 ตรวจความพร้อมหน้างาน 8 ข้อ** (พื้น/ผนัง/ฝ้า/ไฟ/น้ำ/ประตู/defect/แบบ 3D final) → กดอนุมัติเริ่มงาน — ระบบแนบ **snapshot สถานะ T0 ณ วินาทีนั้นลง audit** (ข้อไหนยังไม่พร้อมแล้วหัวหน้าตัดสินใจเริ่ม = มีบันทึกความรับผิดชอบ ไม่ใช่การ block แบบตายตัว เพราะบางข้อ conditional ตามบ้าน)

### 5.4 งานรายวัน — ช่างไม่ต้องเรียนรู้อะไรเลย

- ช่างทำงานตามเลนตัวเอง จบเลน → **ถ่ายรูป Wrapping ส่งในกลุ่มทีมตามปกติ** → bot เก็บเข้าระบบเป็นหลักฐานรายห้องอัตโนมัติ + ตอบรับ (ระบบดึงไฟล์จริงจาก LINE เข้าคลังของเราภายใน 5 นาที — ก่อน LINE ลบ)
- เจอปัญหา → พิมพ์ `#ปัญหา ท่อครัวรั่ว` → ระบบเปิดใบปัญหาผูกบ้าน + **push แจ้งหัวหน้างานทันที** (ข้าม quiet hours ได้เพราะเป็นความรับผิดชอบส่วนตัว) — หลักฐานไม่หายในแชท
- แชทคุยเล่นทั่วไป → **ระบบไม่เก็บ** (PDPA — เก็บเฉพาะ รูป + #ปัญหา + เข้า/ออกกลุ่ม)
- เน็ตหน้างานหลุด (PWA) → รูป/รายงานเข้า**คิวออฟไลน์** ส่งเองเมื่อเน็ตกลับ — ช่างเห็นแค่ "ค้างส่ง N รายการ" ระบบกันส่งซ้ำด้วย id ต่อใบ

### 5.5 ปิดบ้าน — จุดที่ระบบเข้มที่สุด

ห้องเสร็จ = 3 เลนติ๊กครบ + รูป Wrapping → ทุกห้องเสร็จ → **หัวหน้าทีมส่ง "ใบปิดบ้าน"** (ใบเดียวต่อบ้าน):

- ระบบเตือนถ้ามีห้องที่ยังไม่มีรูป (soft — บ้านที่มีห้องยกเลิกกลางทาง หัวหน้าใช้ดุลยพินิจได้ + ลง audit)
- **คนอื่นส่งแทนไม่ได้**: ระบบตรวจกับ RACI ว่าผู้ส่งเป็นหัวหน้าทีมติดตั้งจริง (ADR-039 — office มีสิทธิ์ทั้งระบบก็ปิดงานติดตั้งแทนไม่ได้)
- ผ่าน → work item ปิด → 🎉 ข้อความยินดีเข้ากลุ่มทีมอัตโนมัติ + แจ้ง Sale/PM

## 6. ส่งมอบ — ลูกค้ากดปุ่มเดียว

หัวหน้า/office กด "ส่งตรวจรับ" → บ้านเข้าสถานะ **customer_review** → **การ์ด Flex เข้ากลุ่มลูกค้า**:

```
┌─────────────────────────────┐
│  ตรวจรับงานติดตั้ง 🏠        │
│  บ้านคุณสมชาย รามอินทรา      │
│  งานเสร็จเรียบร้อยแล้วครับ    │
│  รบกวนตรวจรับงานด้วยครับ     │
│ ┌───────────┐ ┌───────────┐ │
│ │ รับงานครับ ✅│ │ ขอแก้ไขก่อน │ │
│ └───────────┘ └───────────┘ │
└─────────────────────────────┘
```

| ลูกค้ากด | เกิดอะไร |
|----------|----------|
| **รับงานครับ ✅** | โปรเจกต์ → **completed** · bot ขอบคุณลูกค้าในกลุ่ม · แจ้งผลเข้ากลุ่มทีมอัตโนมัติ · audit ปิดเส้น 🏠 |
| **ขอแก้ไขก่อน** | โปรเจกต์คง customer_review · แจ้งทีมทันที → รายการเก็บงาน (punch list) → แก้เสร็จส่งตรวจรับใหม่ — **งานหลักไม่ถูก reopen** (นาฬิกาทีมกับนาฬิการอลูกค้าแยกกัน) |

ความปลอดภัยที่มองไม่เห็น: การ์ดทุกใบมี **secret token เฉพาะใบ** (ปลอม postback ไม่ได้) · กดได้เฉพาะในกลุ่มของบ้านนั้น (ระบบตรวจกลุ่ม↔บ้านตรงกัน) · กดซ้ำ/การ์ดเก่า = ระบบรับทราบแต่ไม่เปลี่ยนผล · ส่งตรวจรับซ้ำ = ใบเดิม ไม่ spam ลูกค้า

## 7. สิ่งที่ลูกค้าไม่เคยเห็น (แต่คือเหตุผลที่ทุกอย่างข้างบนง่าย)

- **Audit ทุกก้าว** — ใครทำอะไรที่ไหนเมื่อไหร่ ตั้งแต่ add OA จนกดรับงาน ลบ/แก้ย้อนหลังไม่ได้ (append-only บังคับที่ DB)
- **เอกสารที่เซ็นแล้วแก้ไม่ได้** — รายงานที่เซ็น, ผลอนุมัติ, template ที่ publish = immutable ที่ระดับ trigger
- **ทุกข้อความขาออกคือ template ที่ผ่านการอนุมัติ** — ไม่มี free-text จาก bot, น้ำเสียงไทยอบอุ่นทุกข้อความ, ไม่มีศัพท์ระบบหลุดถึงผู้ใช้
- **PDPA ทั้งเส้น**: consent ตอนผูกตัวตน · แชทกลุ่มไม่ถูกดักเก็บ · ข้อมูลบุคคลอยู่จุดเดียวที่คุมสิทธิ์แน่น
- **ทุก retry idempotent** — webhook ซ้ำ, กดปุ่มซ้ำ, เน็ตหลุดส่งซ้ำ — ผลลัพธ์เกิดครั้งเดียวเสมอ
