# /grill-me — Grill Owner Decisions จาก Ground Truth ของ Repo

Session คั้นมติเจ้าของ (คุณเดฟ) แบบ**คำถามแข็ง ไม่มี softball** — สแกน repo หา decision
ที่ค้างจริง แล้วไล่ถามทีละข้อจน "ตอบแล้วเดินงานได้" จบด้วยการบันทึกเป็น ADR
ตาม house format เดิม (ตัวอย่างผลลัพธ์จริง: ADR-034/035 "grill owner-decisions",
ADR-066/067/068 "มติ owner ก x4")

> ใช้เมื่อไม่มีเอกสารตั้งต้นเฉพาะ — ถ้ามีเอกสาร/รายงานให้ grill จากมัน ใช้ `/grill-with-docs`

## กติกาเหล็ก (ห้ามละเมิด)

1. **AI = advisory เท่านั้น** — เสนอทางเลือก + คำแนะนำ + trade-off ได้ แต่**มติเป็นของ owner
   คนเดียว** ห้ามบันทึกสิ่งที่ owner ไม่ได้ตอบชัดเจน ห้ามตีความคำตอบกำกวมเป็นมติ
2. **Verify-don't-trust ก่อนถามเสมอ** — ทุกคำถามต้องยืนบน ground truth ที่ตรวจแล้วจาก
   repo จริง (ไฟล์:บรรทัด, สถานะ task, ADR ที่มีอยู่) — บทเรียน ADR-034: เอกสารเก่าบอก
   "open decision" ทั้งที่ตัดสินไปแล้ว → **เช็ค ADR log ก่อนถามซ้ำทุกครั้ง**
3. **หนึ่งคำถาม = หนึ่ง decision** — ไม่ยัดหลายมติในคำถามเดียว; เรียงตาม dependency
   (ตัวที่ blocking งานมากสุดก่อน) ไม่ใช่ตามความรุนแรง
4. **ห้ามกุตัวเลข** — เป้าหมาย/ราคา/กำหนดการที่ไม่มีหลักฐานต้นทาง เสนอเป็น "กรอบวิธีคิด"
   หรือ "ต้องเก็บ baseline ก่อน" เท่านั้น (กติกาเดิมของทีม: ห้ามตั้งเป้าจากอากาศ)
5. **แยกชนิดข้อความ** ทุกคำถาม: [หลักฐาน] (อ้างไฟล์ได้) / [การตีความ] / [ข้อเสนอ] —
   ห้ามให้ของจริงปนกับข้อเสนอ

## Steps

### 1. Harvest — สแกนหา decision ที่ค้างจริง

ไล่จากแหล่งเหล่านี้ (grep + อ่าน ไม่เดา):

```
.kiro/specs/*/tasks.md          → "รอ owner" / "owner decision" / "รอมติ" / "- [ ]" ที่ gate อยู่
.kiro/steering/architecture-decisions.md → Consequences ที่เปิดงานค้าง / "ยังไม่ implement"
docs/governance/                → sign-off ที่ยังไม่ครบ / CT-DEC ที่รอ ratify
docs/*scrutiny*.md              → findings ที่ระบุว่าต้องมติ owner
git log + PR ที่เปิดค้าง        → merge ที่รอ owner approve
```

### 2. Verify — ตัดคำถามผี

ต่อ candidate แต่ละตัว: เช็คว่า**ยังเปิดจริง** — ค้น ADR log + โค้ด + spec ว่ายังไม่ถูกตัดสิน/
ทำไปแล้ว ถ้าถูกตัดสินแล้วให้ตัดทิ้งและบันทึกไว้ในรายงานเปิด session ว่า "เคยถูกถามแล้ว → ADR-xxx"

### 3. Rank — เรียงตาม blocking impact

จัดลำดับ: (1) hard gates ที่ block งานหลายสาย (2) decision ที่ block track เดียว
(3) นโยบาย/ทิศทางที่ไม่ block โค้ดวันนี้ — แสดงรายการทั้งหมดให้ owner เห็นก่อนเริ่ม
พร้อมบอกว่าจะถามกี่ข้อ

### 4. Grill Loop — ถามทีละข้อ (Q1..Qn)

ต่อคำถาม ใช้โครงนี้ (ใช้ AskUserQuestion เมื่อมีตัวเลือกชัด):

```
Qn — <ชื่อ decision สั้น>
[หลักฐาน]   สถานะจริงจาก repo (ไฟล์:บรรทัด / run id / ADR ที่เกี่ยว)
[การตีความ] ทำไมต้องตัดสินตอนนี้ + อะไร block อยู่
ทางเลือก    ก) ... (trade-off)   ข) ... (trade-off)   [ค) ...]
[ข้อเสนอ]   แนะนำข้อไหน เพราะอะไร — และผลถ้าเลื่อน (cost of delay)
```

- **Hard mode**: ท้าสมมติฐานของ owner ถ้าหลักฐานขัด · ชี้ contradiction ระหว่างเอกสาร/โค้ด
  ตรง ๆ · ถ้าคำตอบกำกวม ("ก็ดีนะ", "เอาไว้ก่อน") ให้ถามซ้ำจนได้มติที่ operational
  (ใคร-ทำอะไร-เมื่อไร-เงื่อนไขอะไร) หรือ owner สั่ง "เลื่อน" อย่างชัดแจ้ง — การเลื่อนก็เป็นมติ
- คำตอบสั้นแบบบ้านนี้ใช้ได้: "ก", "ก x4", "ตามคำแนะนำทุกข้อ"

### 5. Record — บันทึกเป็น ADR (append-only)

ต่อมติที่ได้ ให้ append เข้า `.kiro/steering/architecture-decisions.md` เลขถัดไป
ตาม house format เป๊ะ:

```markdown
## ADR-0XX — <ชื่อมติ> (<วันที่>, มติ owner <ก/ข...>)

- **Status:** Accepted (grill-me <วันที่>) [— **ยังไม่ implement** ถ้ากระทบโค้ดที่ยังไม่แก้]
- **Context (verified):** <ground truth ที่ตรวจแล้ว พร้อมไฟล์อ้างอิง — เฉพาะของจริง>
- **Decision (owner, <คำตอบตามจริง เช่น ก>):** <มติเต็ม — quote คำตอบจริงของ owner>
- **Consequences:** <งานที่ปลดล็อก / spec ที่ต้อง rebase / งานใหม่ที่เกิด>
```

- อัปเดต quote-block ใน `tasks.md` ของ spec ที่ได้รับผลกระทบ (append-only เช่นกัน)
- มติที่กระทบโค้ดที่ยังไม่แก้ → ติดป้าย **"ยังไม่ implement"** เสมอ (ADR-017/018 pattern)
- commit แยกจากงานโค้ด: `docs(adr): ADR-0XX — <หัวข้อ> (grill-me <วันที่>)`

### 6. Close — สรุป session

รายงานท้าย: มติที่ได้ (พร้อมเลข ADR) · งานที่ปลดล็อกทันที · คำถามที่ owner เลื่อน
(พร้อมเหตุผล + จะโผล่อีกเมื่อไร) · คำถามที่ตัดทิ้งเพราะถูกตัดสินแล้ว

## Success Criteria

- ทุกมติที่บันทึก = คำตอบชัดแจ้งของ owner (quote ได้) ไม่ใช่การตีความ
- ทุกคำถามยืนบนหลักฐานที่ verify แล้ว — ศูนย์คำถามผี (ถามซ้ำของที่ตัดสินแล้ว)
- ADR ใหม่ผ่าน format บ้านครบ (Status/Context (verified)/Decision/Consequences)
